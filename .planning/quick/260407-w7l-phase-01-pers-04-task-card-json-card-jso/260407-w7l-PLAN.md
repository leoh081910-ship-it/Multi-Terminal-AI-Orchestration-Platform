# Quick Plan: Phase 01 PERS-04 task card_json source-of-truth fix

## Objective
补上 Phase 01 的 PERS-04 缺口：让 task 的业务字段以 `card_json` 为唯一业务真相。执行代理需要以最小改动修复 `internal/store/repository.go` 与 `internal/server/server.go`，并用聚焦测试证明不会破坏 Phase 1 已有行为，尤其是 retry、stats、列表读取和 wave 相关存储行为。

## Context
- 验证缺口来自 `E:\04-Claude\Projects\多终端 AI 编排平台\.planning\phases\01-foundation\01-VERIFICATION.md`：当前 `card_json` 已持久化，但读写仍依赖重复结构化列，不满足 PERS-04。
- 当前 repository 在 `CreateTask`/`UpdateTask` 中同时写结构化列和 `card_json`；当前 server 在 task 读接口中直接返回 `ent.Task`。
- 本次 quick 只修复 task card source-of-truth，不引入 delete API，不扩展 schema，不改 wave/event 设计。

  <name>Task 1: 收敛 repository 写入语义为“card_json -> 结构化列派生”</name>
  <files>internal/store/repository.go, internal/store/repository_test.go</files>
  <behavior>
    - Test 1: CreateTask 接收的 card 中如果业务字段与 card_json 冲突，最终落库的 dispatch_ref/state/retry_count/loop_iteration_count/transport/wave/topo_rank/workspace_path/artifact_path/last_error_reason 等结构化列必须与 card_json 一致，而不是与外层重复字段一致。

    - Test 3: 若 card_json 为空或非法 JSON，CreateTask/UpdateTask 返回明确错误，不写入部分脏数据。
    - Test 4: 为兼容已有 Phase 1 行为，state/retry_count/transport 等运行时字段在后续 UpdateTaskState 后仍可按结构化列被查询到。

  <action>在 `internal/store/repository.go` 中新增或内聚一个专用转换路径：先以 `card.CardJSON` 反序列化出规范 TaskCard 业务对象，再从该对象派生所有结构化列并写入 ent。禁止继续把 HTTP 解码后的外层重复字段直接当作最终写入源。CreateTask 仍保持初始创建语义，但其结构化列必须来自 `card_json` 派生值；若现有约定要求首次创建固定 state 为 queued，则在 action 内显式区分“业务 card 真相”和“运行时状态字段”如何兼容，并在测试里锁死预期，避免破坏 retry/state machine。UpdateTask 同样先以 card_json 为准刷新业务字段，并保留必要的 `updated_at` 更新。测试覆盖 create/update/invalid-card_json 三类路径，避免只测 happy path。</action>

  <done>`repository.go` 对 task 的业务写入不再依赖重复列输入；结构化列始终由 `card_json` 派生；非法 `card_json` 会被拒绝；既有状态迁移持久化能力保持可用。</done>
